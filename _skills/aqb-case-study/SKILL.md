# AQB Case Study Builder

This skill produces the full deliverable chain for adding a case study to angieqbailey.com:

1. **Case study page** — dark-variant branded HTML (the main narrative)
2. **Branded artifact pages** — dark-variant HTML for each diagnostic tool/framework produced
3. **Blank downloadable templates** — ecru light-variant HTML for reusable frameworks (print-ready)
4. **Warehouse card** — the HTML snippet for the main site's Warehouse section

The design system is called "Durable Births" (dark variant). The site now uses a shared stylesheet (`/styles.css`) for the index, verdict essays, and L&D pages. Case study pages and artifact pages still use self-contained inline CSS (this works and doesn't need to change, but new shared components should go in styles.css).

## Site Architecture Context

The index page has 5 numbered sections plus an unnumbered Debrief:
- **Section 01:** The Investigation (about)
- **Section 02:** The Warehouse (8 case study cards, collapsible on hover)
- **Section 03:** Research & Development (R&D) — Hardwiring Dignity / DRS
- **Section 04:** Learning & Development (L&D) — links to ld/infrastructure-of-a-life.html
- **Section 05:** The Verdict (5 clickable principles linking to essay pages in verdict/)
- The Debrief (contact, no section number)

Nav links on ALL pages: Investigation | Warehouse | R&D | L&D | Verdict | Debrief

Case studies live in The Warehouse (Section 02). When adding a new case study, the Warehouse card goes on the index and the case study page goes in `posts/`.

Warehouse cards on the index now collapse their `.card-text` and `.card-metric` by default and unfurl on hover. When adding a new Warehouse card, the text/metric will be hidden until hover — keep descriptions concise since they only appear on interaction.

### MANDATORY: GA4 Tracking Tag

**Every HTML page generated by this skill MUST include the Google Analytics 4 tag immediately after the opening `<head>` tag.** This applies to case study pages, branded artifact pages, AND blank downloadable templates. No exceptions. The tag is already in the design-system.md boilerplate — do not remove it. If generating any HTML `<head>` section, verify the GA4 snippet is present:

```html
<!-- Google tag (gtag.js) -->
<script async src="https://www.googletagmanager.com/gtag/js?id=G-S0MXDDKP4E"></script>
<script>
  window.dataLayer = window.dataLayer || [];
  function gtag(){dataLayer.push(arguments);}
  gtag('js', new Date());
  gtag('config', 'G-S0MXDDKP4E');
</script>
```

### MANDATORY: Dark-Variant Page Setup

**Every dark-variant HTML page (case study pages AND branded artifact pages) MUST include these three elements to prevent white flash on page load:**

1. Inline background on `<html>`: `<html lang="en" style="background-color:#121A28;">`
2. Color-scheme meta tag after charset: `<meta name="color-scheme" content="dark">`
3. Inline background on `<body>`: `<body style="background-color: #121A28;">`

The `color-scheme` meta tells the browser to default to a dark canvas before CSS loads. The inline styles provide fallback. All three are required because different browsers respect different signals.

**Critical:** Artifact card links on case study pages must NOT use `target="_blank"`. Same-tab navigation inherits the dark background during page transition. New-tab links flash white because the browser paints its default canvas before HTML loads.

### MANDATORY: Check Existing Pages First

Before building any new page or fixing any rendering issue, fetch 1-2 working pages from the repo via the GitHub API and compare their HTML structure, link attributes and CSS patterns. Match the working pattern before inventing new solutions.

## The Interview

Before writing any code, gather these inputs. Some Angie will provide upfront; others you'll need to ask about.

### Required Inputs

- **Case name**: What should this case study be called? (e.g., "E-Commerce Strategic Summit")
- **Category tag**: What type of work? (e.g., "Strategic Operations," "Organizational Design," "Process Architecture")
- **Cross-functional tag**: Secondary descriptor (e.g., "Cross-Functional Alignment," "Decision Architecture")
- **The situation**: What was broken? What couldn't the organization do? Write this as a forensic finding, not a sales pitch.
- **The diagnostic**: What did the pressure test reveal? List specific constraints with Red/Yellow severity flags.
- **The approach**: What structural intervention was designed? (days/phases, core principle)
- **The artifacts**: What diagnostic tools and frameworks were produced? For each:
  - Name (e.g., "Expectations Stress Test")
  - Type (Framework, Applied Example, Diagnostic Tool, Methodology, Register, Scorecard, Roadmap)
  - Description (2-3 sentences, forensic tone)
  - Whether it gets a blank downloadable template (most do, but not all)
- **The outcome**: Measurable results using the **Scope + Output** pattern:
  - Scope: how big was the mess (stakeholders, domains, functions)
  - Output: what tangible thing was produced (roadmap, decision architecture, execution plan)
  - Specific numbers: initiatives, pressure points documented, timeline locked, etc.
- **Source material**: Raw artifacts, notes, diagnostic outputs to be anonymized and formatted

### Decision Points

These vary per case study — ask if not obvious:

1. **Section arc**: The default is Header → Evidence → Situation → Diagnostic → Approach → Outcome. Evidence (Frameworks & Artifacts) goes right after the case lede so readers see the deliverables above the fold. Some case studies may skip the Diagnostic section if findings are embedded elsewhere, or add a "Structural Context" section before the Situation. Ask if the default works.
2. **Artifact types**: Not every artifact will be a table-based framework. Some may be process maps, scorecards, org charts, or narrative methodologies. The visual structure of each branded artifact page adapts to its content type.
3. **Template eligibility**: Only artifacts deemed reusable as blank frameworks get downloadable templates. Applied examples (specific to one engagement) typically don't, but generalized frameworks do. Confirm with Angie.
4. **Anonymization level**: Default is to strip all personal names and use role/function titles instead. Company-specific system names (e.g., "Shopify," "ERP") are kept unless Angie says otherwise.

## File Structure

Every case study lives under `Portfolio Work/` in a named directory:

```
Portfolio Work/
  {Case Name}/
    {case-name}.html              ← Main case study page
    assets/
      {artifact-slug}.html        ← Branded artifact pages (dark variant)
      {artifact-slug}.md          ← Sanitized markdown source (optional)
      templates/
        {artifact-slug}-blank.html ← Blank downloadable templates (ecru variant)
```

**Naming conventions:**
- Directory: Title Case with spaces (e.g., `e-Commerce Summit`)
- HTML files: kebab-case (e.g., `e-commerce-summit.html`, `expectations-stress-test-framework.html`)
- Template files: `{artifact-slug}-blank.html`

## Building the Case Study Page

Read `references/design-system.md` for the full HTML/CSS boilerplate. Here's the structural arc:

### Nav Bar
```html
<nav class="nav-bar" role="navigation" aria-label="Primary">
    <a href="/" class="nav-mark" style="text-decoration: none;">AB</a>
    <ul class="nav-links">
        <li><a href="/#investigation">Investigation</a></li>
        <li><a href="/#warehouse">Warehouse</a></li>
        <li><a href="/#rnd">R&D</a></li>
        <li><a href="/#lnd">L&D</a></li>
        <li><a href="/#verdict">Verdict</a></li>
        <li><a href="/#debrief">Debrief</a></li>
    </ul>
</nav>
```

### Header Section
- `case-meta`: "Case Study // {Category Tag}" in IBM Plex Mono
- `case-tag`: "{Cross-Functional Tag}" in Rosso
- `h1`: Case name in Imbue, with one word in Rosso via `.accent-word`
- `case-lede`: 2-3 sentence summary with Rosso left-border. The core tension. Forensic, not aspirational.

### The Evidence (Artifacts) — SECOND SECTION
Place this right after the header, before the Situation. Readers should see the deliverables above the fold.
- Section label: "The Evidence"
- h2: "Frameworks & Artifacts"
- Brief intro paragraph
- `.artifact-grid` with `.artifact-card` links:
  - Each card links to the branded artifact page: `href="assets/{slug}.html"` — **NO `target="_blank"`** (same-tab navigation prevents white flash on dark pages)
  - `.artifact-type`: Framework / Applied Example / Diagnostic Tool / Methodology
  - `.artifact-name`: Short name with `<br>` for line break
  - `.artifact-desc`: 2-3 sentence description
  - `.artifact-action`: "View {Type} →" using `&rarr;` entity

### The Situation
- Section label + h2 in the diagnostic-finding style ("No Process Could Survive the Complexity")
- Body paragraphs (IBM Plex Sans, 300 weight, ecru-dim) describing the structural failure
- Optional: `.failure-loop` element showing the recurring pattern that prevented progress
- Bold key phrases with `<strong>` (renders in full ecru, 500 weight)

### Pressure Points / Diagnostic Findings
- Section label: "Pressure Points Surfaced" or similar
- h2: "What the Diagnostic Found"
- Brief intro paragraph, then `.constraint-list` with `.constraint-item` elements
- Each item: `.constraint-flag` (red or yellow) + `.constraint-label` + `.constraint-detail`
- Show 5-8 key findings; reference the full register as an artifact for completeness

### The Approach
- Section label + h2 describing the structural intervention
- Body paragraph explaining the design philosophy
- `.approach-grid` (2x2 grid) with `.approach-item` cards:
  - `.approach-day`: "Day 01" / "Phase 01" / "Principle" in Rosso
  - `.approach-title`: Short title in Imbue
  - `.approach-desc`: 2-3 sentences explaining the phase
- The fourth card is typically a guiding principle, not a day/phase

### The Outcome
- Section label: "The Outcome"
- h2: "What the {Event} Produced"
- `.outcome-grid` with `.outcome-stat` elements:
  - `.outcome-number`: Big Imbue number (one highlighted in Rosso via `.highlight`)
  - `.outcome-label`: What the number means, in IBM Plex Mono
- Closing paragraph with the "shortest version" summary
- `.back-link`: "Back to The Warehouse" linking to `/`

### Footer
```html
<footer>
    <div class="footer-content">
        <span>&copy; {year} Angie Bailey</span>
    </div>
</footer>
```

### JavaScript
- IntersectionObserver for `.reveal` elements (fade-up on scroll)
- All sections except the header get the `reveal` class

## Building Branded Artifact Pages

These are the dark-variant pages that display the actual diagnostic content. Read `references/design-system.md` for the full boilerplate.

Key differences from the case study page:
- Same design system (dark variant, blueprint overlay, spine, nav)
- Content area uses `.doc-content` with `padding-left: var(--spine-left)`
- Navigation links back to the case study page AND the main site
- Has a download button at the top linking to the blank template (if one exists):
  ```html
  <a href="templates/{slug}-blank.html" class="download-btn">
      View Blank Template &rarr;
  </a>
  ```
- Content structure depends on artifact type:
  - **Table-based** (frameworks, registers, maps): Use `.branded-table` with proper thead/tbody
  - **Card-based** (methodologies, mindsets): Use day/phase cards with descriptions
  - **Grid-based** (comparisons, scorecards): Use `.comparison-grid` or `.impl-grid`
  - **Narrative** (summaries, principles): Use structured sections with callout boxes

### Anonymization Rules
- Replace all personal names with function/role titles
- Replace named real-world events with generic equivalents (e.g., "Hurricane Ian" becomes "active hurricane emergency" or "major weather event") unless Angie confirms they can stay. This applies to natural disasters, lawsuits, product launches, acquisitions and any event that could identify the client.
- Keep system names (Shopify, ERP, Salesforce) unless Angie says to genericize
- Keep department and team names
- Preserve all quantitative data (dollar amounts, percentages, counts)
- Use role-based attribution: "VP of Operations" not "Sarah"

## Building Blank Downloadable Templates

These are ecru (light-variant) pages designed for printing and filling in. Read `references/design-system.md` for the CSS boilerplate.

Key characteristics:
- Ecru (#F5F5DC) background, dark text (#1a1a1a)
- Same three-font system but with light-mode color assignments
- Print button in header: `<button class="print-btn" onclick="window.print()">Print / Save as PDF</button>`
- Button styled: Parisian Night background, Ecru text, IBM Plex Mono, uppercase
- Button hidden in print via `@media print { .print-btn { display: none; } }`
- For table-based templates: `@page { size: landscape; margin: 0.5in; }` and print-optimized styles
- For non-table templates (card-based methodologies): Keep portrait, no `@page` landscape
- Empty rows with `&nbsp;` cells for manual or typed entry
- Flag key showing Red/Yellow/Green severity indicators
- Attribution footer: `{Type} by Angie Bailey // angieqbailey.com`

### Template Structure
```
.template-header
  .template-type     → "Blank Template // {Artifact Type}"
  h1                 → Name with .accent span in Rosso
  .subtitle          → Subtitle in Quetzal Green
  .instructions      → 2-3 sentences on how to use, with <strong> for key phrases
  .print-btn         → Print button

.section
  .section-label     → Section category
  h2                 → Section title
  (flag key if applicable)
  table / day-cards / fields

.attribution         → Credit line
```

### How Many Empty Rows
- Frameworks and registers: 10-12 empty rows
- Friction maps and handoff analyses: 10 empty rows
- Methodologies with day-cards: 3 day structures with blank fields

## Building the Warehouse Card

This HTML gets added to the main site's Warehouse section. The pattern:

```html
<a href="posts/{case-slug}/{case-slug}.html" class="evidence-card" style="text-decoration: none; cursor: pointer;">
    <div class="card-tag">{Category Tag}</div>
    <div class="card-title">{Case Name}<br>{Line 2}</div>
    <div class="card-text">
        {2-3 sentence forensic summary of what was broken and what was built.
         First person. Direct. No marketing language.}
    </div>
    <div class="card-metric">Scope: <span>{scope metrics}</span> // Output: <span>{output metrics}</span></div>
</a>
```

The **Scope + Output** metric line is the signature pattern:
- Scope = how big was the operational terrain (stakeholders, domains, functions, departments)
- Output = what tangible deliverable was produced (roadmap, execution plan, decision architecture)

This pattern holds across all case studies. Even when the "output" is less tangible (cultural shift, decision-making capability), reframe it as the structural artifact that enabled it. "Decision architecture for 3 executive functions" is concrete. "Improved alignment" is not.

## Companion Document

Each case study includes a companion .docx narrative document — a rich prose version of the case study with full diagnostic narrative. This is the 5th deliverable in the chain (after case study page, artifact pages, templates and warehouse card).

Key requirements:
- Follows the same section arc as the HTML page (Evidence, Situation, Pressure Points, Approach, Outcome)
- Extended diagnostic narrative with specific forensic detail
- Each companion doc should showcase different capabilities from other case studies
- Built with docx-js (install via `npm install docx` in a temp directory)
- Brand fonts: Georgia for headlines, Arial for body, Courier New for labels
- Brand colors: Parisian Night (#121A28), Bitter Cacao (#2D1B14), Quetzal (#00685E), Rosso (#C41E3A), Ecru (#F5F5DC)
- Header/footer with Angie Bailey branding and page numbers
- Validate with `python scripts/office/validate.py`

## Quality Checklist

Before delivering any case study page set, verify:

1. **Brand compliance**: All palette colors used correctly. Quetzal Bright (#00A896) for green text on dark. No pure black or white. No rounded corners. No italics.
2. **Typography**: Imbue for headlines, IBM Plex Sans for body, IBM Plex Mono for labels/metadata. All mono text at 0.85rem minimum. Correct weights.
3. **Section order**: Header → Evidence → Situation → Diagnostic → Approach → Outcome. Evidence goes above the fold.
4. **Spacing**: Section padding at 2.5rem top/bottom (not 5rem). Tighter sections prevent "is the page over?" scroll fatigue.
5. **Card backgrounds**: `rgba(245, 245, 220, 0.14)` for approach items, artifact cards, outcome stats. Not `var(--ecru-ghost)`.
6. **Voice**: Forensic, architectural tone. No banned words. No marketing-speak. No em dashes (use ellipses). No Oxford commas.
7. **Links**: Artifact card links on case study pages use same-tab navigation (NO `target="_blank"`) to prevent white flash on dark-variant pages. The "View Blank Template" button on artifact pages opens the template in the same tab.
8. **Templates**: Every eligible artifact has a blank template. Templates have print buttons. Table-based templates use landscape `@page`.
9. **Anonymization**: No personal names in any published artifact. Role/function titles only. No named real-world events (hurricanes, lawsuits, product launches) unless Angie confirms they can stay.
10. **Scope + Output**: Warehouse card metric line uses the pattern with specific numbers.
11. **Footer**: Copyright with current year. No taglines.
12. **Responsive**: All pages work at mobile breakpoints (900px and 500px). Nav links drop to 0.75rem at 500px.
13. **Scroll reveal**: All content sections (except header) have the `reveal` class and IntersectionObserver JS.
14. **Nav links**: All 6 nav items present (Investigation, Warehouse, R&D, L&D, Verdict, Debrief). Links use `/#section-id` format for subpages.
15. **Deploy safety**: Always pull latest index.html from repo before editing. Use deploy.js for all pushes. Never hardcode PATs in files pushed to the repo.
16. **Shared stylesheet**: New CSS components that apply across page types should be added to `/styles.css`, not inline. Case study pages keep inline CSS for now.

## Deploying to GitHub Pages

The site is hosted on GitHub Pages from the `AngieQBailey/angieqbailey` repo.

### Preferred Method: deploy.js

A safe deploy script lives in the repo root (`deploy.js`). It always fetches the latest file version from GitHub before pushing, preventing overwrites of manual edits.

Usage from the sandbox:
```bash
GITHUB_PAT="<ask Angie for current PAT>" node deploy.js <local-file> <repo-path> "<commit message>"
```

Examples:
```bash
GITHUB_PAT="github_pat_..." node deploy.js index.html index.html "Add new Warehouse card"
GITHUB_PAT="github_pat_..." node deploy.js "Portfolio Work/My Case/my-case.html" "posts/my-case/my-case.html" "Deploy case study"
GITHUB_PAT="github_pat_..." node deploy.js styles.css styles.css "Add new CSS component"
```

The script will:
1. Fetch the latest version from GitHub (if file exists)
2. Compare file sizes for verification
3. Push with correct SHA to prevent conflicts
4. Report success/failure with new SHA

**CRITICAL: Always fetch the latest index.html from the repo before editing it locally.** Angie makes copy edits directly on GitHub. If you edit a stale local copy and deploy, her edits will be overwritten. This has happened before. The deploy.js script prevents SHA conflicts but does NOT prevent content loss if you never pulled the latest version.

To pull latest before editing:
```bash
curl -s -H "Authorization: Bearer $GITHUB_PAT" -H "Accept: application/vnd.github.v3+json" \
  "https://api.github.com/repos/AngieQBailey/angieqbailey/contents/index.html" | \
  python3 -c "import json,sys,base64; data=json.load(sys.stdin); \
  open('index.html','w').write(base64.b64decode(data['content']).decode('utf-8'))"
```

### Fallback: Direct Node.js API calls

If deploy.js is not available in the workspace, use direct Node.js `https` module calls to api.github.com. The PAT must be provided by Angie at the start of the session — do NOT hardcode PATs in any file pushed to the repo (GitHub's secret scanning will block the push).

## Post-Deployment Documentation (REQUIRED)

After all case study files are uploaded to the repo, complete these three documentation updates before closing the task. These are not optional — they close the loop so the repo and the Website & Case Studies Project stay in sync.

### 1. Update `_changelog.md` in the repo

Fetch the current `_changelog.md` from the repo, append a new entry at the top of the log, and upload the updated file. Entry format:

```
### {YYYY-MM-DD} — {Case Name}
- Added: `posts/{case-slug}/{case-slug}.html` (case study page)
- Added: `posts/{case-slug}/assets/{artifact-slug}.html` (one line per artifact page)
- Added: `posts/{case-slug}/assets/templates/{artifact-slug}-blank.html` (one line per template)
- Added: Warehouse card to `index.html`
- Category: {Category Tag}
- Artifacts: {count} branded pages, {count} blank templates
```

### 2. Update `_site-registry.md` in the repo

Fetch the current `_site-registry.md` from the repo and add the new case study entry with all its pages and artifact paths. Follow whatever format the registry already uses — do not invent a new structure. Upload the updated file.

### 3. Update `site-map.md` in the Project folder

Update the local `site-map.md` file in the Website & Case Studies Project folder. Add the new case study page (and any artifact/template pages) to the "Live Pages" table. Move any matching entry from "Planned Pages" to the table. Update the "Last Updated" column with today's date.

This file is the Project's local memory of what's on the site. If it falls out of sync with the repo, Claude will make incorrect assumptions about what's already published in future sessions.

### 4. Update memory files

Update relevant memory files in `/sessions/.auto-memory/`:
- `project_site_architecture.md` if the index structure changed
- `MEMORY.md` index if new memory files were created
- Create new project memory files for significant new content
